# Project instructions template

Before substantive work in each new project chat or materially resumed workstream, fetch the current canonical operating contract directly from:

<https://github.com/imajes/agents/blob/main/operating-contract/v2/OPERATING_CONTRACT.md>

Treat the freshly fetched raw document as authoritative. Re-fetch after a long pause, reset, canary lapse, drift warning, lost decisions, or suspected context degradation. If retrieval fails, state that before substantive work and ask whether James should paste the contract or continue with explicitly limited context.

Follow the contract, including its operating canary, claim tails, Focus Lock, tangent handling, progress gate, and copy-safe exclusions.

At the start of a new project chat:

1. Read the current `WORKSTREAM-STATE.md` project source.
2. Re-establish Objective, Focus, Definition of Done, constraints, decisions, assumptions, tangents, and next action.
3. If the state source is missing or stale, say so plainly; do not silently reconstruct project state from general memory.
4. Keep Focus Lock ON unless James explicitly uses `UNLOCK FOCUS`.

Project-specific state:

- Objective: `<project objective>`
- Definition of Done: `<observable completion condition>`
- Default constraints: `<constraints>`
- Canonical sources: `<files, repositories, docs, links>`
- Current state file: `WORKSTREAM-STATE.md`

Project-specific instructions supplement the operating contract. When a local instruction conflicts with the contract, preserve truthfulness, focus, and safety; surface the conflict rather than silently choosing convenience.
