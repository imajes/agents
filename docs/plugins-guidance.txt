**Assessment**

Plugin support in this codebase is real, but it is still closer to an internal/experimental platform than a polished external plugin SDK.

The strongest support is in the core/app-server pipeline: there is a manifest loader, marketplace discovery, install/uninstall RPCs, cached installation under `CODEX_HOME`, config-backed enablement, plugin mentions in turns, and curated marketplace sync. The clearest sources are [codex-rs/core/src/plugins/manifest.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manifest.rs#L11), [codex-rs/core/src/plugins/marketplace.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/marketplace.rs#L19), [codex-rs/core/src/plugins/manager.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manager.rs#L548), [codex-rs/core/src/plugins/store.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/store.rs#L11), and [codex-rs/app-server/README.md](/Users/james/src/artificial_intelligence/codex/codex-rs/app-server/README.md#L176).

The weak point is authoring/documentation. I did not find normal user-facing plugin docs in `docs/` or the top-level README. The only concrete “how to build one” material is a sample skill bundle under [codex-rs/skills/src/assets/samples/plugin-creator/SKILL.md](/Users/james/src/artificial_intelligence/codex/codex-rs/skills/src/assets/samples/plugin-creator/SKILL.md#L1), its sample spec at [codex-rs/skills/src/assets/samples/plugin-creator/references/plugin-json-spec.md](/Users/james/src/artificial_intelligence/codex/codex-rs/skills/src/assets/samples/plugin-creator/references/plugin-json-spec.md#L1), and the scaffold script at [codex-rs/skills/src/assets/samples/plugin-creator/scripts/create_basic_plugin.py](/Users/james/src/artificial_intelligence/codex/codex-rs/skills/src/assets/samples/plugin-creator/scripts/create_basic_plugin.py#L1). I also did not find any actual checked-in `.codex-plugin/plugin.json` or `.agents/plugins/marketplace.json` in this repo itself, only sample assets and tests.

**How To Build And Register One**

The on-disk contract is:

- Plugin root contains `.codex-plugin/plugin.json`. The runtime manifest loader explicitly reads only `name`, `description`, `skills`, `mcpServers`, `apps`, and `interface`; those are the fields that actually matter at load time. See [codex-rs/core/src/plugins/manifest.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manifest.rs#L13).
- Manifest paths must be relative `./...` paths and stay inside the plugin root. See [codex-rs/core/src/plugins/manifest.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manifest.rs#L269).
- Skills are loaded from the default `skills/` directory plus any manifest `skills` path. MCP servers default to `.mcp.json` unless `mcpServers` overrides it. Apps default to `.app.json` unless `apps` overrides it. See [codex-rs/core/src/plugins/manager.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manager.rs#L1535) and [codex-rs/core/src/plugins/manager.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manager.rs#L1554).
- The sample scaffold matches that layout and can create the optional directories/files. See [codex-rs/skills/src/assets/samples/plugin-creator/scripts/create_basic_plugin.py](/Users/james/src/artificial_intelligence/codex/codex-rs/skills/src/assets/samples/plugin-creator/scripts/create_basic_plugin.py#L43).

Registration with Codex is marketplace-based, not just “drop a folder somewhere”:

1. Put an entry in `<root>/.agents/plugins/marketplace.json` or `~/.agents/plugins/marketplace.json`.
   The marketplace format and sample entry are documented in [codex-rs/skills/src/assets/samples/plugin-creator/references/plugin-json-spec.md](/Users/james/src/artificial_intelligence/codex/codex-rs/skills/src/assets/samples/plugin-creator/references/plugin-json-spec.md#L98).
2. The marketplace file must live exactly under `<root>/.agents/plugins/marketplace.json`, and plugin `source.path` must be `./...`, resolved relative to `<root>`, not relative to `.agents/plugins/`. See [codex-rs/core/src/plugins/marketplace.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/marketplace.rs#L340) and [codex-rs/core/src/plugins/marketplace.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/marketplace.rs#L383).
3. Codex discovers marketplaces from `HOME/.agents/plugins/marketplace.json` and from repo roots derived from the `cwd` values passed to `plugin/list`. See [codex-rs/core/src/plugins/marketplace.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/marketplace.rs#L284) and [codex-rs/app-server/src/codex_message_processor.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/app-server/src/codex_message_processor.rs#L5790).
4. A marketplace entry only makes the plugin discoverable/listable. Installation is a separate step: `plugin/install` copies the plugin into `CODEX_HOME/plugins/cache/<marketplace>/<plugin>/<version>` and writes `plugins.<plugin>@<marketplace>.enabled = true` to config. See [codex-rs/core/src/plugins/store.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/store.rs#L12), [codex-rs/core/src/plugins/store.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/store.rs#L97), and [codex-rs/core/src/plugins/manager.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manager.rs#L558).
5. At runtime, plugin loading is config-driven: user config has `plugins.<plugin-id>.enabled`, where plugin ids are `<plugin>@<marketplace>`. See [codex-rs/core/src/config/mod.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/config/mod.rs#L1336) and [codex-rs/core/src/config/types.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/config/types.rs#L830).

**Important Gaps / Caveats**

- The plugin RPCs are explicitly marked “under development; do not call from production clients yet” in [codex-rs/app-server/README.md](/Users/james/src/artificial_intelligence/codex/codex-rs/app-server/README.md#L179).
- TUI support exists behind the `plugins` feature and `/plugins` command, but it is not fully symmetric with the backend. For example, plugin tool suggestions are still unavailable in `codex-tui`. See [codex-rs/tui/src/chatwidget/plugins.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/tui/src/chatwidget/plugins.rs#L132) and [codex-rs/core/src/tools/handlers/tool_suggest.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/tools/handlers/tool_suggest.rs#L105).
- The sample spec is broader than the actual runtime parser. The sample includes fields like `version`, `author`, `homepage`, `repository`, `license`, `keywords`, and `hooks`, but the current loader in [codex-rs/core/src/plugins/manifest.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manifest.rs#L13) does not consume most of those. So today those fields are scaffold/documentation only, not strong evidence of implemented behavior.
- Curated plugins are a separate path: Codex syncs `https://github.com/openai/plugins.git` into `CODEX_HOME/.tmp/plugins` and uses that as an extra marketplace root. See [codex-rs/core/src/plugins/startup_sync.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/startup_sync.rs#L27) and [codex-rs/core/src/plugins/manager.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/manager.rs#L1097).
- Discoverability for `tool_suggest` is curated/allowlisted, not general-purpose marketplace discovery. See [codex-rs/core/src/plugins/discoverable.rs](/Users/james/src/artificial_intelligence/codex/codex-rs/core/src/plugins/discoverable.rs#L14).

Bottom line: if you want to build a plugin for this Codex stack today, the practical recipe is “create a plugin folder with `.codex-plugin/plugin.json`, optional `skills/`, `.mcp.json`, `.app.json`; add it to `<root>/.agents/plugins/marketplace.json` or `~/.agents/plugins/marketplace.json`; enable the `plugins` feature; then use `plugin/list` and `plugin/install`.” What is missing is a stable, first-class external plugin guide and a manifest spec that clearly distinguishes implemented fields from aspirational scaffold fields.
